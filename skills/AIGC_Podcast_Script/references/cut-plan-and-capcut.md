# 剪辑决策表 / cut plan / EDL + 剪映落地

SKILL.md Step 7 调用本文件。本文件只回答**"表怎么填、表怎么变成剪映里的东西"**——决策表完整字段释义、与主轨清单的关系、剪映/CapCut 手动拼装 SOP、`pyJianYingDraft` 自动生成草稿的脚本骨架与坑。

**不写**"为什么这时候切"（→ `references/reaction-shot-playbook.md`）、不写片段怎么生成的 prompt（→ `references/prompt-recipes-dialogue.md`）。

> 设计契约：**决策表的每一行 = 上层覆盖轨的一个 cutaway，列与 `pyJianYingDraft` 的 API 参数一一对应。** 一行 → 一段手动操作，也 → 一段生成代码。这是把"拍脑袋插反应镜头 + 手动拼接"变成"照表拼装 / 照表生成"的核心。

---

## 一、两张表的关系

决策表不是一张孤表，它建在 **Step 3 的主轨片段清单**之上。两表分工：

```
主轨片段清单（Step 3 产出）        剪辑决策表 / EDL（Step 7 产出）
─────────────────────────         ─────────────────────────────
主轨骨架：所有说话片段             cutaway 叠加：每行盖一个反应镜头
按时间顺序铺满主轨                 冗余主轨定位列（入点/时长）便于阅读
duration = 素材真实时长            指向主轨某秒 + 指向一条反应素材
        │                                    │
        └──────── 主轨入点(s) 由主轨清单累加 ─┘
```

- **主轨片段清单**回答"主轨上按顺序放哪些说话片段、各多长"。它是连续的、铺满时间线的。
- **剪辑决策表**回答"在主轨的哪个秒、盖哪条静音反应片段、盖多久"。它是稀疏的、只在需要 cutaway 的窗口出现。
- 决策表里的 `主轨入点(s)` / `主轨时长(s)` 是从主轨清单**冗余抄过来**的定位列——抄进来是为了让 cutaway 起点能一眼对照，不用回翻主轨清单。

> 决策表里也可以保留"叠加反应=否"的行来描述纯说话片段（见下方示例第 3 行），这样一张表就能完整还原时间线；也可以只列有 cutaway 的行、把纯说话片段留给主轨清单。两种都行，**整条管道内保持一致即可**。

## 二、决策表 / EDL 完整字段 schema

| 列名 | 含义 | 取值 / 约束 |
|------|------|-------------|
| **序号** | cutaway 行号 | `1, 2, 3 …` |
| **对应台词** | 这个 cutaway 盖在哪句台词上（行号 + 摘要） | 引用 Step 1 行号，如 `L03 A「关键在于复利」` |
| **主轨片段ID** | 被盖的主轨说话片段 | `<人物>_say_<两位>`，如 `A_say_03` |
| **主轨入点(s)** | 该主轨片段在时间线的起点 | 由前序主轨片段时长累加得出；从主轨清单抄入 |
| **主轨时长(s)** | 该主轨片段时长 | **= 素材真实时长**（拿到素材后读真实时长回填，不用估算值） |
| **是否叠加反应** | 该片段上是否盖 cutaway | `是 / 否`（否 = 此行只描述主轨，无覆盖片段） |
| **反应片段ID** | 用哪条反应素材 | `<人物>_react_<token>_<两位>`，如 `B_react_nod_01`；"叠加=否"时留空 |
| **cutaway 主轨起点(s)** | 覆盖片段在**时间线**第几秒开始 | `≥ 主轨入点`，落在句末 / 换气 / 拼接缝；"叠加=否"时留空 |
| **cutaway 时长(s)** | 盖多久 | `0.5–3.0`；遮跳切时 ≥ 接缝两侧各 0.3s（即 ≥ 0.6s） |
| **转场** | cutaway 进出转场 | 默认 `硬切`（= 不加任何 transition） |
| **静音** | 覆盖片段是否静音 | **恒为「是」（硬性字段，不可改）**——盖画面不盖声音的关键 |
| **触发规则** | 命中的规则号（自检用） | `T1–T9`（定义见 `reaction-shot-playbook.md`） |
| **情绪标签** | 该句台词情绪（驱动反应选择，自检用） | `严肃 / 玩笑 / 惊讶 / 质疑 / 中性 …` |
| **备注** | 遮跳切 / loop / 倒放 / 多窗复用等说明 | 自由文本 |

**取值约束补充：**

- ⚠️ **主轨入点 / cutaway 起点用的是「时间线绝对秒」**，不是片段内的相对秒。`cutaway 主轨起点 = 主轨入点 + 该 cutaway 在这段说话片段内的偏移`。例：`A_say_03` 入点 6.0s，想盖在它说到第 3.5s 的位置 → cutaway 起点 = 6.0 + 3.5 = 9.5s。
- ⚠️ **cutaway 不能溢出主轨片段**：`cutaway 起点 + cutaway 时长 ≤ 主轨入点 + 主轨时长`，否则反应镜头会盖到下一句的画面上，语义错乱。
- **静音列永远是「是」**——它在这张表里看着冗余，但保留它是为了对齐 `pyJianYingDraft` 的 `volume=0.0` 参数（一列 → 一个参数）、也提醒手动剪辑时别忘了静音。

## 三、姊妹表：反应素材库需求清单

决策表的 `反应片段ID` 列每一个值，都必须能在这张清单里找到对应行（命名机械对齐）。格式：

| 列名 | 含义 | 取值 / 约束 |
|------|------|-------------|
| **反应ID** | 唯一命名 | `<人物>_react_<token>_<两位>`，如 `B_react_nod_01` |
| **人物** | 谁出现在反应镜头里（= 听者） | `A / B / two`（双人镜头用 `two`） |
| **反应类型** | 反应词表的 token + 中文 | `nod 点头` / `frown 皱眉` / `listen 聆听` …（全表见 playbook） |
| **时长(s)** | 素材时长（决定能盖多长 / 能否循环铺） | 反应片段一般 0.5–3s；idle/listen 类可剪长备循环 |
| **机位** | 固定机位描述 | 单人固定（= 对应单人首帧的机位） |
| **数量** | 同类多备几条防重复 | `1` 或 `2`（高频类型备 2 条） |

**去重逻辑**（决定这张清单有几行）：决策表里同一个 `反应片段ID` 出现多次，清单里只占**一行**——这正是 Step 5"去重归并"的产物。一条 `listen` 在剪映里循环/倒放就能铺满多个时间窗，不必每个 cutaway 单独抽。

## 四、填好的示例表（覆盖纯说话 + 带 cutaway 两种行）

**场景**：A、B 对谈复利话题。主轨已按 `A_say_03 (6.0s起, 7.0s) → B_say_04 (13.0s起, 5.0s) → A_say_05 (18.0s起, 4.0s)` 铺好。

### 剪辑决策表 / EDL

| 序号 | 对应台词 | 主轨片段ID | 主轨入点 | 主轨时长 | 叠加反应 | 反应片段ID | cutaway主轨起点 | cutaway时长 | 转场 | 静音 | 触发规则 | 情绪 | 备注 |
|------|----------|------------|----------|----------|----------|------------|-------------|-------------|------|------|----------|------|------|
| 1 | L03 A「关键在于复利」 | A_say_03 | 6.0 | 7.0 | 是 | B_react_nod_01 | 9.5 | 1.5 | 硬切 | 是 | T2 | 严肃 | 数据落点，切 B 点头 |
| 2 | L04 B「但我未必同意」 | B_say_04 | 13.0 | 5.0 | 是 | A_react_frown_01 | 14.2 | 1.4 | 硬切 | 是 | T4 | 质疑 | 被反驳，切 A 皱眉权衡 |
| 3 | L05 A「哈哈说真的」 | A_say_05 | 18.0 | 4.0 | 否 | — | — | — | 硬切 | — | — | 玩笑 | 纯说话片段，无 cutaway |

> 第 3 行 `叠加反应=否`：此行只是把主轨片段 `A_say_05` 登记进时间线（入点 18.0、时长 4.0），覆盖轨相关列全部留空。它证明决策表也能描述"光放说话片段、不盖反应"的行。

### 配套反应素材库需求清单（上表用到的两条反应素材）

| 反应ID | 人物 | 反应类型 | 时长(s) | 机位 | 数量 |
|--------|------|----------|---------|------|------|
| B_react_nod_01 | B | nod 点头 | 1.5 | 单人固定（`B_solo.png` 机位） | 1 |
| A_react_frown_01 | A | frown 皱眉 | 1.6 | 单人固定（`A_solo.png` 机位） | 1 |

**逐行读法（一行 → 一个动作）**：第 1 行 = "时间线 9.5s 处，把 `B_react_nod_01` 静音盖到上层覆盖轨，盖 1.5s，硬切进出"。这句话既是手动 SOP 的一步，也是一段生成代码——见下两节。

## 五、剪映 / CapCut 手动拼装 SOP（工具中立，人人可用）

这一档不依赖任何库，**任何平台、任何剪映/CapCut 版本都能照表手拼**，是地基。把决策表逐行翻译成剪辑台操作：

### 阶段 A — 铺主轨（用主轨片段清单）

1. **新建项目**，分辨率/帧率与素材一致（如 1920×1080 / 24fps）。
2. 按主轨清单顺序，把所有说话片段（`*_say_*.mp4`）**首尾相接拖到最底层视频轨（主轨）**——它们自带音频+口型，**保持原声、不要静音**。
3. 拖完后核对：每段在时间线上的入点应等于决策表 `主轨入点`，时长应等于 `主轨时长`（= 素材真实时长）。若对不上，多半是某段素材真实时长和表里填的不一致——**以素材真实时长为准，回填决策表**。

### 阶段 B — 叠 cutaway（逐行扫决策表，只处理「叠加反应=是」的行）

对决策表每一个 `叠加反应=是` 的行，做这五步：

4. **新建一条上层视频轨**（剪映里把反应片段往主轨**上方**空白区域一拖，会自动生成"画中画/覆盖轨"；CapCut 同理）。整条片只需一条覆盖轨，所有 cutaway 都放它上面。
5. 把该行 `反应片段ID` 对应的反应片段拖到覆盖轨。
6. **静音这段覆盖片段**：选中它 → 音量调到 0（或点静音）。⚠️ **这一步对应决策表"静音=是"，绝不能漏**——漏了主轨说话声和反应片段的环境音/呼吸声会双声串台。
7. **对齐时间窗**：把这段覆盖片段的**左边缘**拖到时间线 `cutaway主轨起点` 秒；把它的**右边缘**裁到 `cutaway主轨起点 + cutaway时长` 秒（即裁成 `cutaway时长` 那么长）。剪映按住边缘拖动可裁剪；对齐时打开磁吸/吸附更准。
8. **转场 = 硬切**：什么都不加就是硬切。决策表 `转场=硬切` 的行**不要**在 cutaway 进出加任何转场效果。

### 阶段 C — 校验

9. **逐行回放**每个 cutaway 窗口：画面应被反应片段盖住、**声音仍是主轨原声**（盖画面不盖声音）。若 cutaway 期间静音了 → 第 6 步把覆盖片段静音搞错成了把主轨静音，检查是哪条轨被静音。
10. **检查层级**：反应片段必须在主轨**上方**才能盖住主轨画面。若反应片段没显示出来，是它跑到了主轨下层——把它的轨道拖到主轨上方。
11. 时长不够铺满某个窗口时（如想盖 2.5s 但素材只有 1.5s）：剪映里对该覆盖片段**复制接续 / 循环 / 倒放接龙**铺满，不必重抽（备注列标 `loop`/`倒放`）。

> 视觉上这就是 **L-cut 效果**：画面先切走、声音延续。手法理论见 `reaction-shot-playbook.md`。

## 六、（加速档）pyJianYingDraft 自动生成剪映草稿

**定位：主推但有前提。** 决策表每行 → 一段代码，一键生成可直接在剪映打开的草稿，省掉第五节的手拼。**全新生成草稿天然绕开剪映模板加密**（加密只坑"读取已有草稿/模板模式"，本工作流走"从零生成"不受影响）。但它带版本/平台前提，**主流程不依赖它，务必先在本机按第七节清单验证再用**。

### 库与安装

| 平台 | 库 | 安装 | 成熟度 |
|------|----|----|--------|
| 国内剪映（JianyingPro） | `pyJianYingDraft` | `pip install pyJianYingDraft` | 主推、最成熟、活跃维护 |
| 国际 CapCut | `pyCapCut` | `pip install pycapcut` | ⚠️ **实验性、能力略弱、仍在从主库迁移** |

### 草稿默认路径

- **Windows**：`C:\Users\<用户名>\AppData\Local\JianyingPro\User Data\Projects\com.lveditor.draft\`
- **macOS**：`/Users/<用户名>/Movies/JianyingPro/User Data/Projects/com.lveditor.draft/`
- ⚠️ **最稳的找法**：在剪映里打开任一草稿 →「草稿参数」栏看「保存位置」。路径与主数据文件名随版本/平台变化（macOS 历史上有版本叫 `draft_info.json`），一律**以剪映实际显示为准**。

### 脚本骨架（决策表 → 草稿，参数化）

下面这段把"一张决策表"机械落成草稿。**API 形态来自官方 demo 与文档；但凡涉及具体版本行为的，注释里已标 ⚠️ 需本机验证。绝不硬编码 `draft_content.json` 字段，全部通过库操作以规避版本漂移。**

```python
import pyJianYingDraft as draft
from pyJianYingDraft import TrackType, trange

# 1) 草稿文件夹 = 剪映「草稿参数 → 保存位置」那个目录（⚠️ 按本机实际填）
DRAFT_FOLDER = r"/Users/<你>/Movies/JianyingPro/User Data/Projects/com.lveditor.draft"
folder = draft.DraftFolder(DRAFT_FOLDER)
script = folder.create_draft("podcast_ep01", 1920, 1080, allow_replace=True)

# 2) 建两条 video track：main（主轨）在下、cutaway（覆盖轨）在上
#    库文档：relative_index 越大越接近前景（上层），故 cutaway(=2) 盖在 main(=1) 之上；生成后在剪映扫一眼确认
script.add_track(TrackType.video, "main",    relative_index=1)
script.add_track(TrackType.video, "cutaway", relative_index=2)

# 3) 主轨片段清单（= 决策表主轨列）：duration 必须用素材真实时长！
main_clips = [
    # (素材绝对路径,            入点s, 真实时长s)
    ("/abs/path/A_say_03.mp4",   6.0, 7.0),
    ("/abs/path/B_say_04.mp4",  13.0, 5.0),
    ("/abs/path/A_say_05.mp4",  18.0, 4.0),
]
for path, start, dur in main_clips:
    seg = draft.VideoSegment(path, trange(f"{start}s", f"{dur}s"))  # 不传 volume → 保留原声
    script.add_segment(seg, "main")

# 4) cutaway 行（= 决策表「叠加反应=是」的行）：静音 + 精确落点 + 硬切
cutaways = [
    # (反应素材绝对路径,                cutaway主轨起点s, cutaway时长s)
    ("/abs/path/B_react_nod_01.mp4",     9.5, 1.5),
    ("/abs/path/A_react_frown_01.mp4",  14.2, 1.4),
]
for path, start, dur in cutaways:
    seg = draft.VideoSegment(path, trange(f"{start}s", f"{dur}s"), volume=0.0)  # volume=0 → 静音
    script.add_segment(seg, "cutaway")
    # 硬切 = 不调用 add_transition；要别的转场才在这里 add_transition(...)

script.save()
```

**逐参数 ↔ 决策表列对照**（这就是"机械映射"）：

| 决策表列 | 脚本里的位置 |
|----------|--------------|
| 主轨片段ID + 主轨入点 + 主轨时长 | `main_clips` 三元组 + `add_segment(seg, "main")` |
| 主轨保留原声 | `VideoSegment(path, ...)` **不传** `volume` |
| 反应片段ID + cutaway主轨起点 + cutaway时长 | `cutaways` 三元组 |
| 静音=是 | `volume=0.0` |
| 落点精确到秒 | `trange("起点s", "时长s")` |
| 转场=硬切 | **不调** `add_transition` |
| 覆盖轨在主轨上方 | 第二条 track 的 `relative_index` 取更大值（库文档：越大越接近前景/上层） |

### 雷区（写进注意事项）

- ❌ **主轨 `duration` 填估算值而非真实时长** —— 黑帧/截断。脚本里 `main_clips` 的时长必须是素材真实时长（可用 `ffprobe` 读）。
- ❌ **覆盖片段忘传 `volume=0.0`** —— 双声串台。
- ❌ **素材用相对路径 / 跨机移动** —— 草稿引用绝对路径，换机或挪文件夹后需在剪映「重新链接素材」。**素材 + 草稿放固定、稳定的绝对路径目录**。
- ⚠️ **自动导出仅 Windows 支持** —— 自动导出依赖 `uiautomation`，本质 Windows-only（不是平台无关的"剪映 6 及以下"问题）；批量导出功能本身在剪映 5.9 / 6.8 测试通过、**仅支持剪映 6 及以下**（剪映 7+ 隐藏了相关控件）。库 README 明确：**生成的草稿仍需在 Windows 版剪映下导出**。macOS / Linux 能否在本机剪映 GUI 直接手动导出该草稿**未经库验证 → 必须本机自测**，不要默认"打开本机剪映点导出即可"。生成草稿本身（本工作流唯一依赖的能力）跨平台可用。
- ⚠️ **别碰模板模式** —— "读取已有草稿改"会撞剪映 6+ 的加密；本工作流只走"从零 `create_draft`"，不要误入模板/读草稿路径。
- ⚠️ **冷门转场依赖剪映内置资源版本** —— 默认硬切最稳；花哨转场设为可选，缺资源时退回硬切。

## 七、需本机验证清单（用前自查，资料随版本/平台漂移）

凡下列项一律标 ⚠️，**绝不在脚本里写死**，生成后在自己的剪映/CapCut 版本里看一眼：

1. **`relative_index` 上下层方向**：库文档明确"值越大越接近前景（上层）"，覆盖轨取比主轨大的值（脚本 `main=1 / cutaway=2` 正确）。稳妥起见生成后在剪映扫一眼确认即可。
2. ⚠️ **`VideoSegment(volume=0.0)` 对视频轨的实际静音效果**——`AudioSegment` 的 volume 已确认，视频轨某版本行为可能有别，以本机实际播放为准。
3. ⚠️ **草稿默认路径与主数据文件名**（macOS 历史上有 `draft_info.json`）——以剪映「草稿参数 → 保存位置」为准。
4. ⚠️ **新生成草稿是否出现在剪映首页列表**——个别版本需重启剪映刷新。
5. ⚠️ **你的剪映/CapCut 版本号 vs 库测试版本**（库主要在剪映 5.9 测试，6+/7+ 行为可能变）——生成失败先看库 issue / 升级库；**绝不硬编码 JSON 字段**。
6. ⚠️ **导出能力**：自动导出依赖 `uiautomation`、**仅 Windows 支持**（与剪映版本无关）；批量导出功能仅剪映 6 及以下（7+ 控件被隐藏）。库 README 称草稿"仍需在 Windows 版剪映下导出"——macOS 能否本机 GUI 导出需自测，别默认可行。
7. ⚠️ **素材绝对路径**在生成机与打开机必须一致，否则需「重新链接素材」。
8. ⚠️ **`pyCapCut`（国际版）成熟度低于 `pyJianYingDraft`，仍在迁移**——CapCut 用户视为实验性，能力以库当前 README 为准。
