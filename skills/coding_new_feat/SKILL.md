# 新功能开发五步法

## 这个 skill 解决什么问题

大多数人拿到一个新功能需求，会跳过"想清楚"和"拆开"，直接开始写 Prompt 让 AI 一把梭。结果是：每一步反复翻车，最后完整流程永远跑不通，还分不清是哪一步坏了。

这个 skill 把新功能开发压成**一条五步管道**，强制"小步执行、逐步验证"：

```
1. 需求分析  → 搞清楚到底要做什么、不做什么（消除模糊性）
2. 任务拆解  → 把大需求拆成 5-7 个可执行、可验证的小步骤
3. 逐步实现  → 一次只做一步，给精确指令（描述结果 + 约束条件）
4. 每步验证  → 做完一步，验证通过再继续
5. 集成测试  → 全部完成后跑通完整流程
```

核心原则：**永远不要一上来就从子任务 1 写到子任务 7。** 把 AI 当成执行力很强的工程搭档，而不是一次性吐出完整成品的魔法盒子。

## 核心心法（贯穿全流程）

1. **先消除模糊，再动手** —— 需求里藏着大量默认决策。你不说清楚，AI 就替你做，且往往不符合项目现状。
2. **拆到可立即验证为止** —— 判断标准：当前这一步做完，你能不能立刻验证？不能就是拆得不够细。
3. **一次只做一个子任务** —— 做完停下来验证，验证通过才进下一步。
4. **描述结果 + 约束，不写实现步骤** —— Prompt 说"要什么"和"边界在哪"，让 AI 读现有代码保持一致，而不是按它的"通用最佳实践"另起炉灶。
5. **不要回退** —— 验证通过的步骤当作已确认的基础。后面出问题先定位到具体子任务，不要凭直觉把前几步全推翻重来。
6. **知道范围在哪停** —— 新手最常见的不是不会写，而是把不大的需求越做越大。需求分析阶段就钉死边界。

---

## Step 1 — 需求分析：搞清楚"到底要什么"

**目的不是写文档，而是消除模糊性。** 后面每一步的指令都建立在这些明确约束之上。

动手前先让 AI 帮你梳理（先不要写代码）：

```
我要给 <项目> 添加 <功能>。
技术栈是 <栈>。

在开始写代码之前，请帮我梳理：
1. 这个功能涉及哪些核心实体和操作
2. 需要我明确哪些决策点（列出问题清单，我来回答）
3. 和现有系统可能存在哪些集成点

先不要写代码。
```

需要钉死的三类问题：

```
功能边界：
  → 范围到哪为止？哪些明确不做？
  → 各种交互的默认行为是什么（编辑/删除/回复/权限）？
  → 谁能做什么操作？

技术约束：
  → 数据存在哪？用现有库还是新建？
  → 要不要分页 / 实时更新？预期数据量？
  → 沿用现有技术栈，不引入新依赖

和现有系统的关系：
  → 怎么和现有用户体系 / 鉴权关联？
  → UI 放在哪个位置？
  → 现有的 API / 风格约定是什么？
```

**产出**：一句话能说清这个功能**做到哪为止**，以及**明确不做什么**。

> 如果用户的需求已经很清楚，可以快速确认边界后直接进 Step 2，不必硬凑问题。但只要存在"AI 会替我做决定"的模糊点，就必须先问清。涉及多个互斥决策点时用 `AskUserQuestion` 一次性问，不要一个个挤牙膏。

---

## Step 2 — 任务拆解：把大需求拆成小步骤

拆解决定协作效率上限。拆得好每步都顺，拆得差每步都翻车。

**粒度标准**：一个子任务能用 3-5 句话说清楚，AI 输出 **不超过 100 行代码**。超出继续拆；低于这个量（比如"加一个字段"）自己改比写 Prompt 更快。

**拆解顺序**：全栈功能按技术层次**从底向上**——数据层 → 接口层 → 展示层 → 集成。每层输出是下层输入，依赖关系天然清晰。

**每个子任务必须标注三件事**：做什么、看什么上下文、怎么验证。

示例（任务评论功能拆成 7 个子任务）：

```
子任务 1：Comment 表设计
  做什么 → Prisma Schema: id, content, authorId, taskId, parentId(可选), createdAt
  上下文 → 现有 User / Task Schema
  验证   → prisma db push 成功

子任务 2：创建评论 API（POST /api/tasks/:id/comments）
  上下文 → Comment Schema + 鉴权中间件
  验证   → curl 测试，数据库有新记录

子任务 3：查询评论列表 API（GET，返回树形结构）
  验证   → curl 返回正确嵌套 JSON

子任务 4：删除评论 API（DELETE，仅作者）
  验证   → 非作者返回 403

子任务 5：评论输入组件
  上下文 → 创建 API 的接口定义
  验证   → 能输入，提交后调用 API

子任务 6：评论列表组件（树形渲染 + 回复/删除按钮）
  上下文 → 查询 API 返回格式 + 现有 UI 风格
  验证   → 页面展示正确

子任务 7：集成到详情页
  验证   → 完整流程跑通
```

执行顺序严格 1→2→…→N，**每步验证通过才进下一步**。

> review 拆解时重点查两点：**粒度是否均匀**（有的太粗有的太细），**是否遗漏边界情况**（权限校验、错误处理）。漏了的步骤直接新增一个子任务，不要塞进已有任务里。

---

## Step 3 + Step 4 — 逐步实现 & 每步验证（交替进行）

实现和验证是**绑在一起**的：写一个 → 验证一个 → 再写下一个。绝不一次写三个。

**每个子任务正式改代码前的准备动作：**

```
1. 让 AI 先读相关代码（schema、现有路由、鉴权方式、目标页面）
2. 让它复述当前目标，确认理解一致
3. 明确本轮只做这一个子任务（"这次只做 X，不要改前端/别的"）
```

**Prompt 写法：描述结果 + 约束条件，不写实现步骤。**

```
基于刚才创建的 <上一步成果>，实现 <本步目标>。

要求：
- <输入/输出约束>
- 使用现有的 <鉴权 / 组件 / 工具>
- <边界与校验>
- 参考现有 <文件> 的写法风格
```

> "参考现有写法风格"这句很关键——让 AI 读现有代码保持一致。

**只给当前步骤需要的最小上下文。** 上下文太宽泛，AI 会"热心"地优化相邻代码、改到不该改的文件。

### 验证规则（不可跳过）

- **真的验证，不要只信 AI 说"改好了"。** 每个子任务的验证标准在 Step 2 已经写好，照着做。
  - 数据层：`db push` / migration 成功，没改坏现有关系
  - 接口层：接口被**真实调用一次**（curl / Postman / 调试页 / 最小脚本均可）——能收参数、能返回、能处理错误
  - 展示层：层级对不对、交互状态切换对不对、操作后列表/数据有没有正确刷新
- **验证失败当场修复**，不要凭感觉觉得"应该差不多"就往下走。
- **报错直接原样丢给 AI**，它擅长根据报错定位修复。
- **同一个错误反复出现超过 2 次** = Prompt 有歧义，换一种说法说清楚，而不是让它反复重试。

### 上下文管理

一次会话对话太长（超过 ~10 轮）时用 `/compact` 压缩，避免上下文过载导致质量下降。

---

## Step 5 — 集成测试：跑通完整流程

所有子任务完成后，跑一遍端到端流程（发表 → 查看 → 回复 → 删除 之类的完整链路）。

**发现问题时，绝不能笼统说"XX 功能坏了"，必须把问题重新定位回某一个子任务：**

```
发不出去      → 先看创建 API
能发但不显示   → 先看查询 API 或列表组件
点删除没反应   → 先看删除 API 或按钮绑定
```

定位清楚再精确修改那一步，不要回头推翻已验证通过的代码。

## 常见翻车场景与应对

| 场景 | 原因 | 应对 |
|------|------|------|
| AI 用了项目里没有的库 | Prompt 没约束技术栈 | Prompt 里写明，或在 CLAUDE.md 写"不引入新依赖，用现有 XXX" |
| 做到一半发现漏了步骤 | 拆解不完整（很正常） | 当前位置**插入新子任务**，不要塞进已有任务 |
| AI 改了不该改的文件 | 上下文太宽泛 | 每次确认修改范围；已经改了就定位哪些属于本次任务，用 Git 精确回退其余 |
| 前后步骤接口对不上 | 后一步对前一步输出做了不同假设 | 在 Prompt 里**明确引用前一步成果**，如写前端时直接给出 API 返回格式，不让 AI 猜 |

---

## 一句话总结

> 拿到一个新功能后，先问清需求边界，再拆成 5 到 7 个可验证的小任务，然后一次只做一步、做完就验。这套原则不依赖任何工具，工具会换代，"小步执行、逐步验证"不会过时。
