# Bug 修复四步定位法

## 这个 skill 解决什么问题

新功能开发是"从 0 到 1"——你知道要做什么，一步步做出来。Bug 修复方向完全相反：**从异常现象倒推原因**，从模糊到精确。

大多数人用 AI 修 bug 是"哪里坏了改哪里"的直觉式操作：扔一张报错截图,AI 猜一个原因,改了没好,再猜……来回十几轮像在黑暗中摸象。**根本问题是你没给 AI 足够线索,AI 只能猜;猜对是运气,猜错浪费时间。**

这个 skill 把修 bug 压成**一条四步管道**:

```
1. 收集线索  → 拿到完整报错文本 + 复现条件（不是截图）
2. 缩小范围  → 从整个项目缩小到 2-3 个文件
3. 定位根因  → 先分析 WHY 再动手,不只看 WHAT
4. 安全修复  → 最小改动 + 验证 + 防回归
```

> 你的角色是**侦探**,AI 是**取证专家**:你负责指方向、给精确线索,它负责分析细节。线索越精确,定位越准,修复越干净。

## 核心心法（贯穿全流程）

1. **第一步不是改,是收集现场** —— 没搞清问题怎么发生、稳不稳定、报错在哪就让 AI 改,本质还是猜。
2. **先复现一遍** —— 确认问题还在、是必现还是偶现、属于哪类(前端报错/接口报错/没报错但结果不对)。
3. **事实和猜测分开** —— 给 AI 已确认的事实("访问 /user/123 白屏 + 这段报错"),不要把你的猜测("我感觉是字段返回错了")当事实喂进去。
4. **先诊断,不要直接改** —— 每次让 AI 先分析原因,你判断方向对不对,再决定下一步。要的是诊断不是处方。
5. **最小改动,不顺手重构** —— AI 修 bug 爱"顺手优化",改得越多引入新 bug 风险越大。先把问题修对,再单独考虑要不要优化。
6. **一次只修一个 bug** —— 修完验证,再修下一个。不要让 AI 同时处理三个不相关的问题。
7. **修根因不是藏症状** —— 看到 `undefined` 就加 `?.`、用 `as` / `@ts-ignore` 压掉类型错,都是把问题藏起来,不是修复。

## Step 1 — 收集线索：给 AI 当好"报案人"

**修 bug 的第一步不是"赶紧改",而是"先把现场信息收集完整"。**

动手分析前,先**自己手动复现一遍**,确认三件事:

```
1. 这个问题现在还存在吗?     → 别分析一个已经消失的问题
2. 必现还是偶现?            → 必现适合直接定位;偶现优先找触发条件
3. 是前端报错 / 接口报错 / 还是"没报错但结果不对"?
                            → 不同类型排查路径不同
```

一份合格的 bug 报告包含四要素:

```
1. 错误信息(完整的,含 stack trace)
   → 整段原样复制,不要自己删减你以为没用的部分

2. 复现条件
   → 访问 /user/123 时必现,部分用户正常

3. 预期行为 vs 实际行为
   → 预期:正常展示用户名;实际:白屏 + 控制台报错

4. 最近的相关改动
   → 昨天更新了用户 API 的返回格式
```

喂给 AI 时,**最后一句一定要写"先分析,不要修改"**:

```
项目在访问用户主页时报以下错误:
<完整 stack trace>

复现条件:访问 /user/123 时必现,部分用户正常。
昨天更新了用户 API 的返回格式。

请先分析可能的原因,不要直接修改代码。
```

> **快捷技巧——"昨天还好今天突然坏":** 最有价值的线索是 git。让 AI 执行 `git log --oneline -10` / `git diff HEAD~3`,或直接让它用中文总结"最近哪些文件最可能和这个 bug 有关"。**最近改动往往是高价值线索。**

## Step 2 — 缩小范围：从项目到几行代码

**核心原则:不要让 AI 在整个项目里大海捞针。** Stack trace 已经给了文件名和行号,顺着调用链往上追即可——**绝大多数 bug 的根因距离报错位置不超过 2-3 个文件。**

新手最容易犯的两个相反错误:① 一上来把整个项目丢给 AI;② 只盯着报错那一行,不看上下游。正确做法介于两者之间:**先看报错点,再看它上游 1-2 层的数据来源。**

### 有 stack trace（runtime error）

让 AI 只看相关的 2-3 个文件,理清这条链路:

```
1. 报错组件本身    → 报错那一行到底在读什么变量?
2. 这个变量从哪来  → props? hook? 接口请求? context?
3. 上游数据长什么样 → API 返回格式有没有变?
```

```
请查看:
1. src/components/user-profile.tsx(报错组件)
2. src/app/api/user/[id]/route.ts(用户 API)

对比 API 返回格式和组件的数据使用方式,
告诉我具体是哪个字段导致了问题。
```

### 无 stack trace（如"按钮点了没反应"）

这类问题要先回答一个更基础的问题:**到底是哪一步没执行?**

```
1. 从 UI 入手    → "这个按钮在哪个组件里?"
2. 顺事件链追踪  → onClick → 调用的函数 → 调用的 API
3. 关键节点加日志 → 让 AI 在 handleSubmit 开头、API 调用前、
                   API 返回后各加一个 console.log,跑一遍看断在哪
```

日志的作用不是"瞎打几个 console.log",而是把链路切成几段,确认断在哪段:点击事件没触发 / 触发了但参数不对 / 参数对了但请求没发出 / 请求发了但失败 / 请求成功但状态没更新。

## Step 3 — 定位根因：理解 WHY，不只是 WHAT

AI 给出精确诊断后(例如"API 重构多包了一层 `data`,但组件仍用旧路径读取"),**先别急着让它改**,先问一个关键问题:

```
这个 API 还有其他调用方吗?如果有,它们是否也受同样影响?
```

很多人跳过这步,结果**修了一个 bug 却漏掉三个同样的 bug**。让 AI 搜索整个项目里调用该 API 的地方,统一排查。

**判断该防御还是该追根因**(别看到 `undefined` 就无脑加 `?.`):

- 这个值**本来就可能没有** → 做防御性处理(合理)
- 这个值**按业务本来一定有** → 继续追为什么它没来(否则只是把异常藏起来)

不要把"避免报错"和"修复根因"混为一谈。

---

## Step 4 — 安全修复：最小改动 + 验证

确认影响范围后再让 AI 改,Prompt 里钉死改动边界:

```
请修复 user-profile.tsx 中的数据读取,适配新的 API 返回格式。

要求:
- 只改数据读取部分,不要重构其他代码
- 加上 user 为空时的防御性处理
- 如果有其他文件也在用旧格式读取这个 API,一并修复
```

> **"只改必须改的,不要重构其他代码"是这一步最重要的约束。** 一旦"修 bug + 重构 + 顺手优化"混在一起,你就分不清到底是哪处改动真正修好了问题,验证也会变得极痛苦。

修完**逐项验证**(不要只验原始 bug):

```
1. 原始 bug 修复了吗?      → 复现路径走一遍,正常
2. 相关功能正常吗?         → 其他用到同一 API/逻辑的地方
3. 边界情况处理了吗?       → 数据为空/用户不存在时有合理表现
4. 有没有引入新问题?       → 跑一遍基本功能,确认无新报错
```

> **线上 bug / 核心页面**,多做一步:让 AI 总结①根因是什么 ②改了哪几个文件 ③还有没有类似位置值得排查 ④以后怎么避免同类问题。把一次"被动救火"变成经验积累。

## 不同类型 Bug 的处理策略

| 类型 | 特征 | 策略 |
|------|------|------|
| **Runtime Error** | 有报错 + stack trace | 顺调用链追 2-3 个文件(见 Step 2) |
| **逻辑 Bug** | 结果不对但不报错(最棘手) | 把"错误行为"翻译成**精确的输入输出对比**:"输入 completed,期望 3 条,实际返回 5 条"。让 AI 逐行解释相关函数,关注比较逻辑。常见根因:大小写不一致、用了 `==` 而非 `===`、字段名不匹配 |
| **构建 / 类型错误** | 代码跑不起来,报错精确到行 | AI 的强项,直接扔给它。但**禁止用 `as` 断言或 `@ts-ignore` "修复"**——那是藏问题;若 AI 这么做,追问"有没有不用类型断言的解决方式" |
| **状态不同步** | 接口成功了但页面没更新 | 根因常在前端状态流,不是接口。问:①请求成功后组件状态更新了吗?②更新的是页面真正依赖的那份状态吗?③有没有因缓存/条件渲染/依赖遗漏导致没重渲染? |

> 逻辑 bug 比报错类更难受,因为"代码能跑"会给人系统没坏的错觉。只要结果不对就是 bug。别急着改条件判断,先把**输入、处理中间值、输出**三件事捋清。

## 常见误区

- **只给截图,不给文本。** 有分析价值的是文本(控制台报错、终端输出、网络响应体、复现步骤)。截图可以附,**文本必须有**。
- **描述 bug 而不展示代码。** "点击排序后顺序不对"只能让 AI 猜。让它**直接读代码**,准确度高一个数量级。
- **修了表面症状。** `user.name` 报 undefined 就加 `?.`?问题不是缺可选链,而是数据**为什么**是 undefined。
- **一次修太多 bug。** 三个 bug 上下文和根因都不同,**一次修一个**,修完验证再下一个。
- **不检查是否引入新问题。** 为修 A 改了 B,B 出新 bug。修完一定跑一遍相关功能,不要只验原始 bug。
- **没定位清楚就大面积回退/重写。** 这不是修 bug,是制造新的不确定性。正确顺序永远是:**先定位 → 再小改 → 再验证。**

---

## 一句话总结

> 先复现,再收集文本线索;先缩小范围,再让 AI 分析;先确认根因,再做最小修复。Bug 修复的核心技能不是"让 AI 改代码",而是"给 AI 正确的线索"。
