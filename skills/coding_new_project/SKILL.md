# 从零搭建项目四步法

## 这个 skill 解决什么问题

新功能开发、Bug 修复都是在**已有项目**上工作。从零搭建是完全不同的起点：**你面前是一个空目录。**

这反而是 AI 最能提效的场景——项目初始化阶段有大量**机械性工作**（建目录、写配置、装依赖、搭样板代码），耗时但不需要创造力。手动可能要半天到一天，AI 帮你半小时搞定。

但有两个最常见的陷阱：

1. **把技术选型全甩给 AI。** 技术选型是架构决策，不是机械劳动。AI 可以帮你调研，但**决策必须你来做**。
2. **把"从零搭建"理解成"一次性把完整产品全做完"。** 第一天就把首页、详情、搜索、标签、评论、登录、后台全塞进去——那不是搭建项目，是在空目录上直接叠复杂度。

这个 skill 把搭建过程压成**一条四步管道**：

```
1. 技术选型   → 决定用什么技术栈（你决策，AI 调研）
2. 项目骨架   → 初始化项目、一次性定好目录结构
3. 工具链配置 → Lint、格式化、Git 配置
4. 首个页面   → 搭出第一个可运行页面，然后写 CLAUDE.md
```

核心策略：**让 AI 干体力活，你做脑力活。** 初始化、配置、样板代码交给 AI；技术选型、架构决策、目录规划你来。

## 核心心法（贯穿全流程）

1. **决策是你的，调研交给 AI** —— AI 只给"训练数据里最流行的那套"，不一定适合你的场景、背景和部署环境。
2. **先求可运行，不求完整** —— 目标是"最小可继续迭代的版本"，不是"完整、炫、一步到位"。
3. **每一步都验证再往下叠** —— 先把项目跑起来，再叠配置，任何一步出错都知道卡在哪。
4. **新手优先选已掌握 60% 以上的技术栈** —— 别在一天里同时学新框架 + 新部署 + 新内容系统 + 新 UI，那是先把自己堆进认知负担。
5. **部署模型反过来约束项目结构** —— 先想清楚"纯静态站点还是需要服务端能力"，再定目录和功能方案。
6. **CLAUDE.md 不早写也不漏写** —— 首个页面完成、项目稳定后立即写。

> 下文以"从零搭建一个个人技术博客"为例走一遍。具体技术栈（Next.js / Astro / Biome…）只是示例，换成任何栈，四步法不变。

---

## Step 1 — 技术选型：决策是你的，调研交给 AI

第一步不是写代码，而是想清楚用什么技术。

**错误用法**（直接甩给 AI 决策）：

```
我想做一个技术博客，帮我选技术栈
```

AI 会给你一个"通用最佳实践"方案，但它不了解你的技术背景、部署环境、团队偏好。**选错了，后面所有工作都建立在错误基础上。**

**正确用法：你列约束 + 候选方案 + 对比维度，AI 做调研对比：**

```
我要搭建一个个人技术博客，需求如下：
- 主要发布技术文章（Markdown/MDX）
- 需要代码高亮和数学公式支持
- 部署到 GitHub Pages（静态站点）
- 我熟悉 React 和 TypeScript
- 希望后续能加交互式代码演示

请对比以下方案：
1. Next.js 静态导出  2. Astro  3. Gatsby  4. VitePress

从【构建速度 / MDX 支持 / 生态丰富度 / 学习成本】四个维度对比。
先不要写代码。
```

关键点：**你给了约束、列了候选、指定了维度。** AI 的任务是调研对比，不是替你决策。

AI 输出对比后，**你**根据自己情况选。比如已经熟 React，Next.js 学习成本最低；内容为主不需要太多交互，Astro 可能更合适。

> 新手实用原则：**优先选你已掌握 60% 以上的栈，而不是"社区现在最火的"。** 最怕的不是框架普通，而是你同时在学太多新东西。

**选定后立刻记录下来**（后面要写进 CLAUDE.md）：

```
技术栈确认：
- 框架：Next.js 14 App Router + 静态导出
- 样式：Tailwind CSS v4
- 内容：MDX + remark/rehype 插件
- 部署：GitHub Pages
- 包管理：pnpm
```

**产出**：一份白纸黑字的技术栈清单，以及你为什么这么选的理由。

## Step 2 — 项目骨架：让 AI 搭结构，一次性定好

技术选型定了，开始搭建。这一步**强烈推荐用 Claude Code**——它直接在终端执行命令，从 init 到配置一条龙。

**关键：不要把"创建项目"和"把项目配完整"混成一个超大任务。** 更稳妥的顺序：

```
1. 先创建项目，确认能启动
2. 再安装内容相关依赖
3. 再配置静态导出 / MDX
4. 再补目录结构和首批页面
```

先把项目跑起来，再往上叠配置。**把边界说清楚**，例如"先只初始化，不要顺手生成一堆业务页面"：

```
请帮我初始化一个 Next.js 技术博客项目。

技术栈：
- Next.js 14 App Router，配置静态导出
- TypeScript strict 模式
- Tailwind CSS v4
- MDX 支持（@next/mdx）
- pnpm

请执行：
1. 用 create-next-app 初始化
2. 安装必要依赖
3. 配置 next.config.ts（静态导出、MDX）
4. 配置 Tailwind

先只执行初始化，完成后告诉我，不要顺手生成业务页面。
```

AI 自己执行命令、装依赖、改配置，**你只需确认每一步输出没报错**。

初始化通过后，**一次性定好目录结构**：

```
src/
  app/              页面路由
    page.tsx        首页
    blog/
      page.tsx      博客列表页
      [slug]/
        page.tsx    根据 slug 读取内容
    about/page.tsx  关于页
  components/
    ui/             通用 UI 组件
    blog/           博客相关组件
  lib/
    utils.ts        工具函数
    posts.ts        文章读取与查询
content/
  posts/            MDX 文章存放目录
```

> **静态导出的坑（新手尤其常见）：** 选了 Next.js 静态导出 + GitHub Pages，博客详情页**不要**写成 `app/blog/[slug]/page.mdx` 这种"看起来自动生效"的结构。更稳妥的做法是：文章放 `content/posts/*.mdx`，在 `app/blog/[slug]/page.tsx` 里按 slug 读取，静态导出时用 `generateStaticParams` 预生成所有路径。**先用最稳妥、最可解释的结构，别追求"看起来很酷的目录写法"。**

**关键原则：目录结构在这一步就定好，后面不要频繁调整。** 目录是项目骨架，频繁改动会引发大量 import 更新，移动文件的成本不低。

**验证**：`pnpm dev`，确认首页能正常访问。

---

## Step 3 — 工具链配置：最枯燥但最重要的一步

项目能跑了，配工具链。这是最枯燥的环节——也是很多人跳过、后来又后悔的环节。

> **跳过的代价：** 等项目有 50 个文件再配 ESLint，你会面对几百个 lint 错误、五花八门的格式、混乱的 import。**在只有几个文件时配好，后续每个新文件都自动遵守规范。**

```
请为项目配置工具链：

1. 代码规范（ESLint + Prettier，或 Biome 二选一）
   - React/TypeScript 规则、import 排序
   - 加 lint / format 脚本到 package.json

2. Git 配置
   - .gitignore（Node.js + Next.js）
   - .editorconfig

3. VS Code 配置（可选）
   - .vscode/settings.json（保存时自动格式化）
   - .vscode/extensions.json（推荐扩展）

逐项配置，每配完一项告诉我，我验证后再配下一项。
```

**两个要点：**

1. **不要同时上 ESLint + Prettier + Biome 三套。** 选一套主方案：要传统生态用 `ESLint + Prettier`，要省配置用 `Biome`。工具越多不是越专业，而是越容易互相打架。**在 Prompt 里明确你的选择**，不要让 AI 自作主张。
2. **配完一定要验证。** 工具链配置格式经常变（ESLint flat config、Prettier v3…），AI 训练数据可能不是最新的。跑一下 lint 看有没有报错，格式化一下看效果对不对。

## Step 4 — 首个页面 + CLAUDE.md

骨架搭好、工具链配好，现在搭第一个**真正有内容**的页面——这标志着从"项目配置"进入"功能开发"。

> 首个页面的意义不只是"做个首页"，而是**用一个真实页面验证前面的技术选型到底顺不顺手**。很多问题（样式方案不顺手、MDX 处理链冲突、暗色模式接入别扭、目录和页面组织不匹配）只有写真实页面才会暴露。

```
请实现博客首页 src/app/page.tsx。

功能：
- 顶部导航栏（首页、博客、关于）
- Hero 区域（标题 + 一句话描述）
- 最近文章列表（暂时用 mock 数据，3 篇）
- 底部 Footer

约束：
- 使用 Tailwind CSS
- 响应式（移动端适配）
- 暗色模式支持
- 参考简洁的技术博客风格
```

首页做完后，你对技术栈、目录结构、编码风格都有了明确认知——**现在是写 CLAUDE.md 的最佳时机。**

```
请基于当前项目的实际情况，生成一份 CLAUDE.md。

包含：
1. 项目概述（一句话）
2. 常用命令（dev、build、lint、format）
3. 目录结构说明
4. 技术约束（框架、CSS 方案、包管理器等）
5. 编码规范（文件命名、组件写法）

控制在 100 行以内，只写最关键的信息。
```

> **为什么不在初始化时就写 CLAUDE.md？** 那时目录可能调整、编码风格还没形成、工具链还没配好，写出来不准确，反而误导 AI。等首个页面做完、这些都稳定了，写出来的才准确。

**到这一步，你就进入了正常的功能开发节奏——也就是 `coding_new_feat` 的五步法。** 后续每个功能（列表页、详情页、搜索…）都按"需求分析 → 任务拆解 → 逐步实现 → 每步验证 → 集成测试"推进。

> **新手在这里先暂停一下**，别立刻继续堆功能。先确认三件事：① 项目能稳定启动和构建吗？② 目录结构你自己能讲清楚吗？③ 现在让 AI 接手，它能从 CLAUDE.md 理解项目吗？三件事都没问题，再往下做。

## 常见翻车场景与应对

| 场景 | 原因 | 应对 |
|------|------|------|
| 让 AI 做所有决策（"你觉得用啥好就用啥"） | 把架构决策当机械劳动 | **决策你做，调研交给 AI**：列约束、列候选、指定维度让它对比 |
| 一步到位、不做验证 | 一次给 AI 超大任务 | **按步骤来**：初始化 → 工具链 → 首页，每步验证通过再进下一步 |
| 跳过工具链配置 | "功能先做出来，工具链以后再说" | 文件还少时配好，每个新文件自动遵守规范 |
| CLAUDE.md 写太早或不写 | 太早内容不准误导 AI，不写则反复解释 | **首个页面完成后立即写** |
| 静态导出和动态能力没分清 | 说要部署 GitHub Pages，又默认想用运行时接口/数据库/登录态 | **部署模型反过来约束结构**：先定"纯静态还是要服务端"，再定目录和功能 |
| 同时学太多新东西 | 框架/部署/内容系统/UI 全是新的 | 优先选已掌握 60% 以上的栈，一次只引入一个陌生项 |

---

## 一句话总结

> 面对空目录，先**你自己**定技术栈（AI 只做调研对比），再让 AI 一条龙搭骨架、配工具链、出首个页面，每步验证通过再往下叠。核心是**先求"可运行、可解释、可继续扩展"，不要一开始就追求"完整、炫、一步到位"**。搭到能跑、能讲清、能从 CLAUDE.md 接手，就切回 `coding_new_feat` 的正常开发节奏。
